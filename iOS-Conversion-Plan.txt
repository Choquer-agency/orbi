ORBI MAIL: iOS APP CONVERSION PLAN
====================================

CONTEXT
-------
Orbi Mail is currently an Electron + React desktop app. The goal is to create an iOS app
that users can download on Apple Silicon Macs via the App Store. The app should feel native
to iOS -- proper app icon, real-time updates, haptic feedback, push notifications, iOS
navigation patterns, and seamless integration with the Apple ecosystem.

APPROACH: CAPACITOR
-------------------
Wraps the existing React frontend in a native iOS shell. This is the right choice because:
- The frontend already has extensive mobile support (MobileBottomNav, MobileComposeSheet,
  swipe gestures, 899px breakpoint, safe area handling)
- Already uses SF Pro fonts, iOS-style tab bar, touch-action: manipulation
- Full rewrite (React Native or SwiftUI) would take months; Capacitor gets a working app in days
- Capacitor bridges native APIs (haptics, push, Keychain, Face ID) directly

TARGET: iOS 14+ (Capacitor 6 minimum), iPhone + iPad + Apple Silicon Mac (via "Designed for iPad")

3C. BACKEND: APNs PUSH SERVICE
--------------------------------
New file: packages/backend/src/services/push-notification.ts
  - Use @parse/node-apn (maintained APNs library, community fork of original apn)
    - Alternative: use APNs HTTP/2 API directly with Node's built-in http2 module
      to avoid the dependency (fewer moving parts, but more code)
  - Initialize provider with .p8 key credentials (token-based auth, NOT certificate-based)
  - Export sendPushNotification(userId, { title, body, data, badge })
  - Query DeviceToken table and send via APNs

APNs environment handling:
  - Development: api.sandbox.push.apple.com (for Xcode debug builds / TestFlight)
  - Production: api.push.apple.com (for App Store builds)
  - Determine from NODE_ENV or a dedicated APNS_PRODUCTION env var
  - IMPORTANT: TestFlight builds use the PRODUCTION APNs environment, not sandbox

Push payload constraints:
  - Maximum 4KB per notification payload
  - Truncate email subject + sender name + snippet to stay under limit
  - Structure: { aps: { alert: { title, body }, badge, sound, thread-id }, data: { threadId } }

Error handling:
  - 410 Gone: device token is invalid/unregistered. DELETE the DeviceToken record immediately.
    Failing to do this causes your token table to grow unbounded with stale tokens.
  - 429 TooManyRequests: implement exponential backoff with jitter (1s, 2s, 4s, max 60s)
  - 403 Forbidden: .p8 key is invalid or expired. Log alert, don't retry.

New env vars in packages/backend/src/config/env.ts:
  - APNS_KEY_ID (optional)
  - APNS_TEAM_ID (optional)
  - APNS_KEY_PATH (optional, path to .p8 file)
  - APNS_BUNDLE_ID (optional, default: com.orbimail.app)
  - APNS_PRODUCTION (optional, boolean, default: false)

3D. BACKEND: TRIGGER PUSH FROM WORKERS
----------------------------------------
Update workers that create notifications:
  - packages/backend/src/workers/gmail-sync.worker.ts
  - packages/backend/src/workers/microsoft-sync.worker.ts

After Socket.io emit, also call sendPushNotification() with:
  - Title: sender name
  - Body: email subject snippet (truncated to ~100 chars)
  - Data: { threadId } for deep navigation
  - Badge: current unread count
  - thread-id: thread ID (for iOS notification grouping -- groups notifications per thread)

Batching strategy:
  - If a sync pulls multiple new emails at once (e.g., first sync, catching up after offline),
    do NOT fire individual notifications for each email
  - Batch into a single summary notification: "5 new emails from 3 senders"
  - Threshold: if more than 3 new emails in a single sync cycle, use summary
  - Below threshold: send individual notifications with thread-id grouping

Silent push for background sync:
  - In addition to visible notifications, send content-available: 1 silent pushes
    to trigger background data refresh (badge update, cache warming)
  - Silent pushes have a lower priority and are throttled by iOS (~2-3 per hour)

3H. BACKGROUND APP REFRESH
----------------------------
Update Info.plist: UIBackgroundModes: ['fetch', 'remote-notification']
  - On background fetch: hit /api/notifications/unread-count, update badge
  - iOS controls frequency (~15-30 min for actively used apps, hours for rarely used ones)

Important caveats:
  - iOS aggressively throttles background fetch for apps that aren't used frequently.
    An app opened once a week may get background fetch opportunities only every few hours.
  - Set expectations: badge counts may be stale until a push notification arrives.
  - Prefer APNs content-available silent pushes over background fetch for badge updates.
    Silent pushes are more reliable and immediate (triggered by your backend, not iOS scheduler).
  - Register for background fetch as a fallback, but don't rely on it as the primary mechanism.

4D. SHARE EXTENSION (SEND CONTENT TO ORBI)
--------------------------------------------
  - Native iOS Share Extension in Xcode project
  - Receives shared text/links/images from other apps
  - Passes to main app via App Groups shared container
  - Main app opens compose with pre-filled content

NOTE: This is the most complex item in the entire plan. A Share Extension is:
  - A SEPARATE Xcode target with its own build settings and lifecycle
  - Runs in a SEPARATE process with a 120MB memory limit
  - CANNOT access the main app's WKWebView or Capacitor runtime
  - Needs its own UI (SwiftUI is the best choice -- small, fast, native)
  - Communicates with the main app via App Groups shared container:
    - NSUserDefaults suite (for small data like text/URLs)
    - Shared file container (for images/attachments)
  - This is the ONLY part of the plan that requires native Swift code
  - After saving shared content to App Groups, open the main app via URL scheme
    (com.orbimail.app://compose?shared=true), and the main app reads from the shared container

App Groups setup:
  - Add App Group capability: group.com.orbimail.app
  - Add to both the main app target AND the Share Extension target
  - Configure in both targets' entitlements files

Recommendation: Defer to end of Phase 4 or even post-launch. The core app
experience is complete without it, and it's 2-3 days of pure native Swift work.

PrivacyInfo.xcprivacy manifest:
  CRITICAL since iOS 17. Apple REJECTS apps that use required-reason APIs without declaring them.
  You must audit every framework and Capacitor plugin for API usage:

  Required API declarations (likely needed):
    - NSPrivacyAccessedAPICategoryUserDefaults
      Reason: CA92.1 (access user defaults to read/write app configuration)
    - NSPrivacyAccessedAPICategoryFileTimestamp
      Reason: DDA9.1 (access file timestamps for cache management)
    - NSPrivacyAccessedAPICategoryDiskSpace
      Reason: E174.1 (check available disk space -- some Capacitor plugins do this)
    - NSPrivacyAccessedAPICategorySystemBootTime
      Reason: 35F9.1 (measure elapsed time -- some libraries use this for performance)

  Audit process:
    1. Build the app in Xcode
    2. Run: Product > Analyze
    3. Check the Privacy Report for any undeclared API usage
    4. Also check each Capacitor plugin's documentation for required privacy declarations
    5. Merge all required declarations into a single PrivacyInfo.xcprivacy file

5I. APP REVIEW PREPARATION
-----------------------------
  Common rejection reasons for email clients and how to prevent them:

  1. Crash on no network: Test with airplane mode. Show offline UI (2K), don't crash.
  2. Incomplete information: Fill out ALL App Store Connect fields. Missing fields = rejection.
  3. Broken login: Ensure demo account works. Test OAuth flows with real credentials.
  4. Private API usage: Audit all Capacitor plugins. Run Xcode static analyzer.
     Use: nm -u YourApp | grep _OBJC to check for private API symbols.
  5. Missing purpose strings: Every capability needs a human-readable usage description:
     - NSFaceIDUsageDescription (Face ID)
     - NSCameraUsageDescription (if using camera for attachments)
     - NSPhotoLibraryUsageDescription (if accessing photos)
     - NSContactsUsageDescription (if accessing device contacts)
  6. Rate limiting on OAuth: If Apple reviewer can't log in because of Google/Microsoft
     OAuth rate limits, they'll reject. Test the OAuth flow works from a fresh device.
  7. No account deletion: Must offer in-app account deletion (5D).
  8. Guideline 4.2 (Minimum Functionality): Apple may question whether a WebView-based
     email client provides enough native value. Mitigate by:
     - Implementing haptics, push notifications, Face ID, and other native features
     - Making sure the app uses native UI elements (share sheet, action sheet, date picker)
     - Not showing a web browser chrome or URL bar

  App Review notes field (max 500 characters):
    Include: demo credentials, any special instructions, explanation of native features.
    Example: "This is a native email client built with Capacitor. Native features include:
    push notifications, Face ID lock, haptic feedback, iOS share sheet, Keychain storage,
    and Spotlight indexing. Demo: review@orbi.agency / [password]"

5K. APP SIZE AUDIT
--------------------
  - Apple enforces a 200MB over-the-air download limit
    (users on cellular can't download apps larger than 200MB)
  - Capacitor apps can get large due to bundled web assets

  Size audit checklist:
    - Build the release IPA and check the size
    - Use Xcode's App Thinning report (Product > Archive > Distribute > App Thinning)
    - Major size contributors:
      - JavaScript bundle (run 'npx vite build --report' to see bundle analysis)
      - Images/assets in the web bundle
      - Capacitor plugins (each adds native code + frameworks)
      - CocoaPods dependencies
    - Optimization strategies if over budget:
      - Enable Vite's code splitting and tree shaking
      - Compress images, use WebP format
      - Lazy-load non-critical routes
      - Remove unused Capacitor plugins
      - Enable Xcode's "Strip Swift Symbols" build setting

  App Thinning:
    - Bitcode is deprecated (removed in Xcode 14) -- ignore any references to it
    - App Thinning for different device architectures is still active and automatic
    - Verify the Xcode project has "Enable Bitcode" set to NO

5L. CI/CD PIPELINE
--------------------
  While Fastlane is listed as "optional" in the original plan, for any production app
  it should be considered essential:

  Fastlane setup (packages/ios/fastlane/):
    - fastlane match: manages certificates and provisioning profiles in a private Git repo
      - Prevents "it works on my machine" signing issues
      - Required for CI builds where you can't use Xcode automatic signing
    - fastlane gym: builds the IPA with consistent settings
    - fastlane pilot: uploads to TestFlight with changelog
    - fastlane deliver: submits to App Store with metadata and screenshots

  GitHub Actions workflow (.github/workflows/ios-build.yml):
    - Trigger: on tag push (v1.0.0, v1.1.0, etc.)
    - Steps:
      1. Checkout code
      2. Install Node.js dependencies (npm ci)
      3. Build frontend (npm run build:ios)
      4. Sync Capacitor (npx cap sync ios)
      5. Install CocoaPods (cd ios/App && pod install)
      6. Run fastlane match to install signing credentials
      7. Build with fastlane gym
      8. Upload to TestFlight with fastlane pilot
    - Secrets required: MATCH_PASSWORD, APP_STORE_CONNECT_API_KEY, FASTLANE_SESSION
    - Runner: must be macOS (GitHub-hosted macos-latest or self-hosted)

  Alternative: Xcode Cloud
    - Apple's built-in CI/CD, free for 25 compute hours/month
    - Configured entirely in Xcode, no GitHub Actions needed
    - Automatically signs builds using your Apple Developer account
    - Limitations: less flexible than GitHub Actions + Fastlane

KEY RISKS (UPDATED)
================================================================================

1. TipTap in WKWebView -- Rich text editor runs in WebView; keyboard handling needs
   testing. Isolated in ComposeInline.tsx, easy to adapt if issues arise.
   MITIGATION: Test early in Phase 2C. If problematic, fall back to textarea on iOS.

2. SSE Streaming -- AI chat uses fetch + ReadableStream for SSE. WKWebView supports
   Streams API but needs early validation.
   MITIGATION: Test in Phase 1. If broken, fall back to Socket.io-based streaming.

3. IndexedDB Persistence -- Offline cache via idb-keyval works in WKWebView but can
   be cleared by iOS under storage pressure. Auth tokens use Keychain (Phase 3G);
   cache loss is acceptable.
   MITIGATION: Already using Keychain for auth. React Query cache loss is self-healing.

4. Socket.io Background -- iOS suspends WebSocket connections in background. Existing
   reconnection logic in useSocket.ts already handles this.
   MITIGATION: Add explicit reconnect on appStateChange resume event (Phase 1J).

5. App Store Review -- Email clients are a known category. Ensure proper offline states,
   error handling, and no private API usage.
   MITIGATION: Offline UI (2K), demo account (5H), review prep (5I).

6. WKWebView Memory Pressure -- iOS can terminate the WKWebView process silently,
   leaving a blank white screen on resume.
   MITIGATION: Handle webViewTerminated event, reload the app (Phase 1J).

7. OAuth in WKWebView -- Google and Microsoft may block OAuth flows inside WKWebView.
   MITIGATION: Use @capacitor/browser for OAuth flows (Phase 1K).

8. Apple Developer Enrollment Timeline -- Organization enrollment requires D-U-N-S
   verification which takes 2-4 weeks.
   MITIGATION: Start enrollment in parallel with Phase 1 development.

9. AASA Caching -- Apple's CDN caches the apple-app-site-association file aggressively.
   Changes during development may not reflect for 24-48 hours.
   MITIGATION: Use AASA validator and swcutil during development (Phase 4B).

10. Share Extension Complexity -- The only component requiring native Swift code.
    120MB memory limit, separate process, no access to main app's WebView.
    MITIGATION: Defer to end of Phase 4 or post-launch (Phase 4D).
